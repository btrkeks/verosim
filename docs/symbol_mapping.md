# MEI → OMR-NED Symbol Mapping

Status: v1 finalized. The authoritative counting semantics are the vendored musicdiff's
(`AnnNote.notation_size`, annotation.py:264; `AnnExtra.notation_size`,
annotation.py:717) per D12; this table maps Verovio's parse-only Object tree
onto them. Empirical findings below were probed with `verosim --dump-tree`
(which also prints the score's scoreDef subtree) against both
importers; HAND-10 (`corpora/hand10/`) pins the counts end-to-end.

## Tree shape (parse-only, breaks:none)

```
Doc → pages → page → {mdiv, score, system, pageMilestoneEnd}   (milestones, flat)
score (milestone)   → m_scoreDefSubtree (NOT a child): scoreDef → staffGrp → staffDef@n
                       with clef / keySig(+keyAccid) / meterSig as CHILD elements
system              → {section (milestone), scoreDef (mid-score changes), measure, ...}
measure             → staff@n → layer@n → events; control elements (tie @startid/@endid);
                       mNum[@type=autogenerated] (ignored)
```

- Mid-score signature changes are `scoreDef` children of `system`, in document
  order before the measure they affect. Components can sit at scoreDef level
  (→ all staves) or inside `staffGrp → staffDef@n` (→ that staff). Both occur
  in one element (kern `*k[b-]` + `*M3/4` probe: keySig at scoreDef level,
  meterSig inside staffDef).
- Layer `@n` values are arbitrary (lieder shows n=5, n=6); document order is
  the voice order. Staff `@n` keys into the part list built from the initial
  scoreDef's staffDefs (document order). A grand-staff instrument is a nested
  `staffGrp` with one `staffDef` per staff — musicdiff's part = one staff
  (music21 PartStaff), so SymPart per staffDef.

## Per-element mapping (Tier A)

| MEI (Verovio class) | SymScore | Symbols counted |
|---|---|---|
| `note` (in layer) | SymNote, 1 pitch | pitch 1 + visible accid 0/1 + head 1 + dots n + beams/flags + tuplet entries + grace 1(+slash 1) |
| `chord` | one SymNote per member note, diatonic-ascending order; carrier (chord) supplies dur/dots/beams/tuplets/offset | per member: pitch + head + dots; beams/tuplets repeated per member |
| `rest` | SymNote, pitch "R" | head 1 + dots n (+ beam entries when beamed/flagged: enhanced-beam rules) |
| `mRest` | SymNote "R"; duration = governing meter's measure length | head (+ dot if meter length needs one, e.g. 3/4 → dotted half) |
| `multiRest` | SymNote "R"; treated as one full-measure rest with a warning (v1 corpus does not require multi-measure expansion) | head (+ dot if meter length needs one, like `mRest`) |
| `space` (hidden, e.g. kern `4ryy`) | nothing (advances offset cursor) | 0 — musicdiff skips `hideObjectOnPrint` notes |
| `beam` (container) | per-note beam-type lists, then musicdiff `get_enhance_beamings` fixups | 1 per beam/flag entry |
| `tuplet` (container, @num @numbase) | per-note tuplet-type entries + tuplet_info (num string on start, "" otherwise) | 2 per active tuplet per note (type + info) |
| `clef` (staffDef child or inline in layer) | SymExtra kind=clef, symbolic `{shape}{line}[{±8}]` | 1 |
| `keySig` (@sig, staffDef child or scoreDef child) | SymExtra kind=keysig, infodict per accidental; sig=0 → `flats/sharps: none` | max(1, |accidentals|) |
| `meterSig` (@count @unit @sym @form) | SymExtra kind=timesig | 2 (numerator+denominator), or 1 (@sym common/cut; @form="num" = MusicXML single-number → numerator only) |
| `mNum`, `stem`, `flag`, `dots`, `label`, `instrDef`, `pgHead`, `grpSym`, `mdiv`, milestones | ignored | 0 |

## Accidentals / effective pitch (D13)

- Every pitched kern note gets an `accid` child; **only the `@accid` value
  counts** as a visible accidental (empty element = none). Maps to musicdiff
  accidental names: `s/f/n/x or ss/ff` → sharp/flat/natural/double-sharp/double-flat.
- Sounding alter: `@accid.ges` when present (MusicXML carried/keysig-implied
  notes get it: iomusxml.cpp m_currentAccids; kern sets it on some paths),
  else resolver: key signature + within-measure carry + tie propagation.
  Ties come from `<tie>` control elements (`@startid` registers the carry,
  `@endid` marks the target) — the element lives in the measure where the
  tie *starts*, so the id sets persist across barlines and are consumed as
  the referenced notes resolve. Neither importer emits the `@tie` attribute
  (checked during validation review, 2026-06-12); the attribute path exists for MEI input
  only. Pinned by `tests/fixtures/tie_chain.krn` against music21.
- MusicXML `<alter>` **without** `<accidental>` element: music21 leaves the
  accidental invisible (HAND-10 tiny.xml pins this) — consistent with @accid
  counting. An `<alter>` contradicting both keysig and carry without a printed
  accidental is unrepresentable on the Verovio side (no @accid, and
  @accid.ges covers it only when iomusxml's carry logic fires) — accepted D6
  parser difference, quantified in the DEV-200 validation audit.
- Kern in-key sharps (keysig-governed) have empty `accid` → 0 symbols
  (HAND-10 keysig.krn); explicit redundant kern naturals keep `@accid="n"` →
  1 symbol (D14).

## Grace notes

| Source | Verovio | musicdiff (graceType, slash) |
|---|---|---|
| kern `qq` | `@grace="acc"` | ('acc', False) |
| kern `q` | `@grace="unacc"` | ('unacc', True) — slash implied, no @stem.mod |
| xml `slash="yes"` | `@grace="unacc"` + `@stem.mod="1slash"` | ('unacc', True) |
| xml no slash attr | `@grace="acc"` | ('unacc', True) — music21 GraceDuration defaults slash=True |
| xml `slash="no"` | `@grace="acc"` (indistinguishable from absent) | ('acc', False) — **unrepresentable**, accepted D6 edge |

Extraction rule: `unacc` → ('unacc', slash); `acc` → kern ('acc', no slash),
MusicXML ('unacc', slash). Requires knowing the input format (recorded by
VrvBridge at load).

## Known v1 caveats (documented, not solved)

- Transposing instruments: musicdiff runs `score.toWrittenPitch()`; Verovio
  stores written pitch already, but key-signature context for the resolver is
  the written one — fine for lieder/grandstaff; revisit for band/orchestra.
- humlib vs converter21 visual-accidental analysis may disagree on exotic
  cases; vendored musicdiff wins (D12) — divergences surface in the count
  audit and are listed here as found.
- `quarterLengthToClosestType` edge durations ('complex') — ported from
  music21; only reachable through mRest/multiRest and irregular tuplet math.

## Importer differences found by the DEV-200 Tier-A count audit (D6, accepted)

Validation summary in `docs/validation.md`; the audit lands at 0.30% (xml) /
1.53% (kern) aggregate |Δ| against the oracle's notation sizes.

1. **converter21 signature restating**: on kern files whose rhythm humlib
   also rejects ("Inconsistent rhythm analysis"), converter21 restates
   clef/keysig/timesig in every measure; Verovio emits them once. Dominates
   the kern-side gap; affected pairs are already ≈0.97 OMR-NED degenerate in
   the oracle itself.
2. **`<time-modification>` without `<tuplet>` notation** (measured tremolo
   halves, unnotated ratios): music21 builds a duration Tuplet (counted
   type + info per note); iomusxml drops the ratio entirely — neither a
   Tuplet container nor `@num/@numbase` on the note survive. Undercount on
   the C++ side AND wrong local offsets for those notes (their @dur ignores
   the ratio) — flagged for matching on affected files.
3. **Cross-staff notes**: music21 and Verovio can assign cross-staff
   beamed/tuplet notes to different staves of a grand-staff part; global
   pitch counts agree, but per-staff beam/tuplet reconstruction shifts.
